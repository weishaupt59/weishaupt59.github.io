▾ES1 Introduzione ai sistemi Unix/LinuxSO, filesystem, shell, comandi base, redirezione

Sistema operativo & storia

  • Timeline: Unix (1969) → BSD (1977→FreeBSD/NetBSD/OpenBSD, Darwin/macOS) · Solaris · GNU (1984) · GNU/Linux (1994 → Debian, Ubuntu, Arch, Gentoo).
  • Componenti GNU/Linux: kernel (controlla l'hardware) + applicazioni (shell / software di sistema / programmi utente).
  • Utenti: root = amministratore (privilegi totali) · utenti normali = privilegi limitati. Buona pratica: account normale per l'uso quotidiano.

Filesystem

  • Gerarchia ad albero, radice unica /. Everything is a file (file, cartelle, dispositivi I/O, rete = stream di byte).

  • Path assoluto: parte da / · relativo: dalla dir corrente.
  • Speciali: ~ home · . dir corrente · .. dir padre. Case-sensitive!
  • Tipo (in ls -l): - file normale, d directory.

Shell & terminale

  • GUI (facile) vs CLI (più potente). Shell testuali: sh csh tcsh bash zsh.
  • Ciclo: prompt → legge comando (Enter) → esegue → stampa output.
  • Prompt: alice@studio:~/Documents$ → utente@host:dir · $ utente normale, # root.
  • Scorciatoie: Tab autocompletamento · ↑↓ history · Ctrl+R ricerca storia · Ctrl+D logout · Ctrl+Alt+T terminale · Ctrl+Alt+F1…F6 terminali virtuali (F7 = GUI).

Permessi speciali SUID / SGID

  • SUID: in esecuzione il processo acquisisce i privilegi del proprietario del file. Simbolo s al posto della x dell'owner — es. -rwsr-xr-x (/usr/bin/passwd).

  • Ottale: cifra extra in testa, 4=SUID, 2=SGID. Es. 6754 → rwsr-sr--.

▾ES5 Processi (I): fork / exec / waitmodello a processi, PCB, creazione e terminazione

Modello a processi

  • OS multiprogrammato a processi, time-sharing. Address space: dati privati, codice condivisibile. Comunicazione via scambio di messaggi.
  • Stati: pronto (ready) · running · sleep/bloccato · zombie (terminato, padre non ha fatto wait) · swapped · init.
  • PCB: Process Structure (sempre in memoria: PID, stato, PPID, priorità, riferimenti a dati/stack/codice) + User Structure (swap-out durante sleep: copia registri, dir corrente, owner, argc/argv…).

Creazione: fork, getpid

pid_t fork(void);     // 0 al figlio, PID del figlio (>0) al padre, <0 errore
pid_t getpid(void);   // PID del processo corrente
pid_t getppid(void);  // PID del padre
  • Il figlio condivide il codice ed eredita una copia di dati globali, stack, heap, User Structure (stesso PC iniziale). Comportamento differenziato dal valore di ritorno.

Terminazione: exit, wait

void  exit(int status);     // terminazione volontaria (status 0-255 al padre)
pid_t wait(int *status);    // padre attende un figlio; ritorna PID o -1
  • wait: sospende se tutti i figli sono in esecuzione · ritorna subito se c'è uno zombie · errore se nessun figlio.
  • Terminazione volontaria (ultima istruzione / exit) o involontaria (azione illegale, es. segmentation fault / segnale).
macro per status — <sys/wait.h>
WIFEXITED(status)    // vero se terminazione volontaria
WEXITSTATUS(status)  // estrae l'exit code

Sostituzione codice: famiglia exec

Sostituisce codice e dati del processo. Lista argomenti terminata da NULL. Non ritorna se ha successo (ritorna solo in caso di errore).

int execl (char* path, char* arg0, ..., (char*)0);
int execv (char* path, char* argv[]);
int execle(char* path, char* arg0, ..., (char*)0, char* envp[]);
int execve(char* path, char* argv[], char* envp[]);
int execlp(char* file, char* arg0, ..., (char*)0);
int execvp(char* file, char* argv[]);

l=lista arg · v=vettore argv · e=environment · p=cerca nel PATH.

▾ES7 Processi (III): gerarchia & job controlinit, ID, RUID/EUID, gruppi, nice, fg/bg, top

Gerarchia & identificatori

  • init: primo processo (PID 1), avviato dal kernel; tutti discendono da lui; adotta gli orfani. Debian/Ubuntu = systemd. Albero: pstree.
pid_t getpid();  getppid();  getpgrp();   // PID, PPID, Process Group ID
uid_t getuid();  getgid();                 // Real UID / GID
uid_t geteuid(); getegid();                // Effective UID / GID
  • RUID/RGID = utente/gruppo che ha lanciato il processo. EUID/EGID = determinano i privilegi di accesso; differiscono dai reali se il comando ha bit SUID/SGID.
  • Invio segnali consentito se l'EUID o RUID del mittente coincide con il RUID del destinatario.

Gruppi di processi & priorità

  • Nuovo processo da terminale → nuovo process group; i figli ereditano il gruppo del padre (preservato da exec). Base del job-control.
  • nice (priorità): intervallo [-20, 19], valore alto = priorità bassa. Root può abbassarla; l'utente può solo aumentarla sui propri processi.

Job control (shell)

  • Un job = un comando/pipeline; ha un Job ID; lista con jobs.
  • Foreground: ha stdin/out/err e il controllo del terminale. Background (cmd &): nessun stdin, torni alla shell.

▾ES8 Thread POSIX (I): pthreads & mutexcreate/exit/join, mutex e sezioni critiche

Concetti

  • Thread = flusso di esecuzione indipendente dentro un processo; più thread condividono risorse e address space ("processi leggeri").
  • Pro: creazione e context switch più leggeri, interazione via risorse comuni. Contro: serve gestire la concorrenza (codice thread-safe, mutua esclusione).
  • Linux: thread nativi a livello kernel = unità di scheduling. Processo tradizionale = thread che non condivide nulla.
libreria & compilazione
#include <pthread.h>
gcc file.c -lpthread            // Debian: -lpthread -std=c99

Identificatori

pthread_t pthread_self(void);                       // ID thread corrente (tipo opaco)
int pthread_equal(pthread_t t1, pthread_t t2);      // confronto
int gettid(void);    // TID Linux-specific (== PID se thread unico) — non portabile

Creazione, terminazione, join

int pthread_create(pthread_t* thread, const pthread_attr_t* attr,
                   void* (*start_routine)(void*), void* arg);  // 0 = ok
void pthread_exit(void* retval);                    // termina il thread
int  pthread_join(pthread_t thread, void** retval); // attende un thread
  • Il programma parte con il main thread (esegue main) che può creare gli altri.
  • Se il main termina senza pthread_exit → i figli muoiono; con pthread_exit → i figli continuano.
  • pthread_join: si blocca finché il thread indicato termina; retval riceve quanto passato a pthread_exit (o NULL).
esempio
void* tr_code(void* arg) {
    printf("Hello World! My arg is %d\n", *(int*)arg);
    free(arg);
    pthread_exit(NULL);
}
int main() {
    pthread_t tr1;
    int* a = malloc(sizeof(int)); *a = 1;
    if (pthread_create(&tr1, NULL, tr_code, a)) exit(-1);
    pthread_exit(NULL);   // così tr1 continua a girare
}
creazione di N thread (loop)
#define NTHREADS 10
int main() {
    pthread_t tr[NTHREADS];
    int* args[NTHREADS];
    int ret;
    for (int i = 0; i < NTHREADS; i++) {
        args[i] = malloc(sizeof(int));
        *args[i] = i;
        ret = pthread_create(&tr[i], NULL, tr_code, args[i]);
        // ... gestione del valore di ritorno ret ...
    }
    pthread_exit(NULL);
}

Mutex (mutua esclusione)

Astrazione tipo semaforo binario: protegge le variabili condivise. pthread_mutex_t rappresenta stato (libero/occupato) + coda di attesa.

pthread_mutex_t M;
int pthread_mutex_init(pthread_mutex_t* M, const pthread_mutexattr_t* attr);
int pthread_mutex_lock(pthread_mutex_t* M);    // wait
int pthread_mutex_unlock(pthread_mutex_t* M);  // signal
pattern sezione critica
pthread_mutex_init(&M, NULL);
pthread_mutex_lock(&M);
/* ... sezione critica: accesso alla risorsa condivisa ... */
pthread_mutex_unlock(&M);

Senza mutex: incrementare una variabile globale da più thread → race condition (risultato non deterministico).

▾ES9 Thread POSIX (II): condition variableswait/signal/broadcast, produttore-consumatore

Variabili condizione

Coda dove i thread si sospendono volontariamente in attesa di una condizione logica (basata su risorse condivise → verificata in mutua esclusione).

pthread_cond_t C;
int pthread_cond_init(pthread_cond_t* C, pthread_cond_attr_t* attr);
int pthread_cond_wait(pthread_cond_t* C, pthread_mutex_t* M); // sospende, rilascia M; al risveglio riacquisisce M
int pthread_cond_signal(pthread_cond_t* C);     // risveglia almeno un thread
int pthread_cond_broadcast(pthread_cond_t* C);  // risveglia tutti
Sempre while, mai if sulla condizione: la signal pthread è "signal & continue" → il risvegliato non acquisisce il lock atomicamente, la condizione va ri-verificata. 
while (condizione_logica)
    pthread_cond_wait(&C, &M);

Esempio: produttore / consumatore

risorsa condivisa (buffer circolare)
typedef struct {
    int buffer[BUFFER_SIZE];
    int readInd, writeInd, cont;   // indici + n. elementi
    pthread_mutex_t M;
    pthread_cond_t FULL, EMPTY;
} risorsa;
risorsa r;
// init: mutex_init(&r.M); cond_init(&r.FULL); cond_init(&r.EMPTY);
//       r.readInd = r.writeInd = r.cont = 0;
consumatore
pthread_mutex_lock(&r.M);
while (r.cont == 0)                       // buffer vuoto?
    pthread_cond_wait(&r.EMPTY, &r.M);
val = r.buffer[r.readInd];
r.cont--;
r.readInd = (r.readInd + 1) % BUFFER_SIZE;
pthread_cond_signal(&r.FULL);            // risveglia un produttore
pthread_mutex_unlock(&r.M);
produttore
pthread_mutex_lock(&r.M);
while (r.cont == BUFFER_SIZE)             // buffer pieno?
    pthread_cond_wait(&r.FULL, &r.M);
r.buffer[r.writeInd] = val;
r.cont++;
r.writeInd = (r.writeInd + 1) % BUFFER_SIZE;
pthread_cond_signal(&r.EMPTY);           // risveglia un consumatore
pthread_mutex_unlock(&r.M);

Esempio: accesso limitato (max MAX_T thread)

#define MAX_T 10
int n_users = 0;
pthread_cond_t FULL; pthread_mutex_t M;

// INGRESSO
pthread_mutex_lock(&M);
while (n_users == MAX_T) pthread_cond_wait(&FULL, &M);
n_users++;
pthread_mutex_unlock(&M);
/* ... uso della risorsa ... */
// USCITA
pthread_mutex_lock(&M);
n_users--;
pthread_cond_signal(&FULL);
pthread_mutex_unlock(&M);

▾ES10 VFS, file descriptor & pipei-node, link, open/read/write, eredità fd con fork, pipe

Filesystem & i-node

  • Tutto è file: ordinario (dati su memoria di massa), speciale (dispositivo), directory (insieme di file).
  • i-node = descrittore del file (tipo, permessi owner/group, dimensione, n. link, vettori di indirizzamento). i-list = lista degli i-node su disco; i-number = identificatore.
  • Directory = record <nome_file, i-number>. L'i-node è eliminato solo quando il n. di hard link = 0.

Strutture dati per l'accesso

  • User Structure (del processo) → Tabella file aperti di processo (i file descriptor) → Tabella file aperti di sistema (un elemento per file aperto, con I/O pointer + riferimento i-node) → Tabella file attivi (i-node in memoria).
  • Ogni file aperto ha un I/O pointer: read/write lo fanno avanzare (accesso sequenziale).
  • Standard fd: STDIN, STDOUT, STDERR aperti automaticamente.
fork & file descriptor: il figlio eredita una copia dei fd che puntano allo stesso elemento della tabella di sistema → padre e figlio condividono l'I/O pointer.

Primitive (non bufferizzate)

int     open(const char* path, int flags);   // ritorna fd (<0 = errore)
ssize_t read (int fd, void* buf, size_t count);        // n. byte letti
ssize_t write(int fd, const void* buf, size_t count);  // n. byte scritti
int     close(int fd);

flags (<fcntl.h>, combinabili con OR): O_RDONLY, O_WRONLY, O_RDWR, O_APPEND (dalla fine), O_CREAT. Lista: man 2 open.

lettura file con buffer fisso
int fd = open(argv[1], O_RDONLY);
if (fd < 0) { perror("Errore open"); exit(-1); }
char buffer[BUF_SIZE];
ssize_t nread;
while ((nread = read(fd, buffer, BUF_SIZE-1)) > 0) {
    buffer[nread] = '\0';
    printf("%s", buffer);
}
close(fd);

La libreria C standard offre l'equivalente bufferizzato: fopen/fread/fwrite/fclose, fscanf/fprintf.

Pipe

  • IPC a comunicazione indiretta (mailbox FIFO), canale monodirezionale con due estremi (lettura/scrittura), ognuno con un fd.
  • Sincronizzazione automatica: il lettore si blocca se la pipe è vuota, lo scrittore se è piena.
  • Con fork i figli ereditano i fd → comunicano con padre/fratelli. (Tra processi non imparentati: socket.)
int pipe(int fd[2]);   // 0 = ok, -1 = errore
// fd[0] = lettura,  fd[1] = scrittura

Standard fd & dup2

  • Macro in <unistd.h>: STDIN_FILENO (0), STDOUT_FILENO (1), STDERR_FILENO (2).
int dup2(int target, int newfd);
// chiude newfd (se aperto) e ne fa una copia di target  → <0 errore

Redirezione: stdout del figlio → pipe

int pipe_fd[2];
pipe(pipe_fd);
if (fork() == 0) {                  // figlio
    close(pipe_fd[0]);              // chiude la lettura
    dup2(pipe_fd[1], STDOUT_FILENO);// stdout → pipe
    execl("/bin/ls", "ls", "-l", NULL);
}
// padre: close(pipe_fd[1]); legge da pipe_fd[0]

Pattern: "pilotare" un'applicazione non modificabile

App che legge da STDIN e scrive su STDOUT. Strategia: 2 pipe + fork + dup2 di stdin/stdout + execl; il padre la guida con write/read.

figlio = l'applicazione
int FtS[2], StF[2];     // father→son, son→father
pipe(FtS); pipe(StF);
if (fork() == 0) {                      // figlio
    close(FtS[1]); close(StF[0]);       // chiude estremi inutilizzati
    dup2(FtS[0], STDIN_FILENO);  close(FtS[0]);   // stdin  ← FtS
    dup2(StF[1], STDOUT_FILENO); close(StF[1]);   // stdout → StF
    execl(path_app, nome_app, NULL);
}
padre = pilota
close(FtS[0]); close(StF[1]);
char sendbuf[1024], recvbuf[1024];
sprintf(sendbuf, "...\n");                       // '\n' = invio!
write(FtS[1], sendbuf, strlen(sendbuf));         // manda al figlio
int nread = read(StF[0], recvbuf, sizeof(recvbuf)-1);  // bloccante
recvbuf[nread] = '\0';
printf("%s", recvbuf);
Il \n è fondamentale (rappresenta l'invio per l'applicazione). Buffer separati per invio/ricezione; read è bloccante. Variante d'esercizio: usare bc al posto dell'app (man bc / bc -h).
